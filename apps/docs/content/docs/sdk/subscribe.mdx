# Subscribe

The `buildSubscribeTransaction` method returns the instructions needed to initialize
an on-chain Subscription PDA and delegate the subscriber's USDC tokens to it. The
caller is responsible for assembling and sending the transaction.

## Method signature

```typescript
const { subscriptionPda, instructions, bump } = recur.buildSubscribeTransaction(
  subscriberWallet,
  options,
);
```

### Parameters

`subscriberWallet: PublicKey` — the subscriber's wallet (must sign the resulting
transaction). Passed positionally as the first argument.

`options: SubscribeOptions`:

| Field | Type | Required | Description |
|---|---|---|---|
| `planId` | `string` | Yes | Plan ID from the Recur API (cuid) |
| `merchantWallet` | `string` | Yes | Merchant's wallet address (base58) |
| `planSeed` | `string` | Yes | 16-char hex string (8 bytes) identifying the plan |
| `amount` | `number` | Yes | Payment amount in USDC base units (6 decimals). `5_000_000` = $5.00 |
| `intervalSeconds` | `number` | Yes | Billing interval in seconds. `2_592_000` = 30 days |
| `delegationCycles` | `number?` | No | How many billing cycles to pre-approve. Default `12` |

The SPL token approve amount is `amount × delegationCycles`. When the delegation is
exhausted, call [`buildReapproveTransaction`](#re-approving-an-exhausted-delegation)
to top it up without touching the on-chain subscription.

### Return value

```typescript
interface SubscribeTransaction {
  subscriptionPda: PublicKey;
  instructions: TransactionInstruction[];
  bump: number;
}
```

| Field | Description |
|---|---|
| `subscriptionPda` | The derived PDA address for this subscription |
| `instructions` | The two raw instructions: SPL approve + `initialize_subscription` |
| `bump` | The PDA bump seed |

The SDK returns raw instructions rather than a `Transaction` object so the caller
can compose them into a versioned transaction, prepend a `ComputeBudget`
instruction, or batch with other operations.

## Transaction contents

The returned `instructions` array always contains exactly **2 instructions**:

### 1. SPL Token Approve

Delegates USDC from the subscriber's Associated Token Account (ATA) to the
Subscription PDA.

- **Amount**: `amount × delegationCycles` (default 12 cycles)
- **Delegate**: the Subscription PDA
- **Owner**: the subscriber (signer)

### 2. Initialize Subscription

Creates the on-chain Subscription PDA with seeds
`["subscription", subscriber, merchant, plan_seed]`.

**Instruction data** (32 bytes):
- Bytes 0–7: Anchor discriminator for `initialize_subscription`
- Bytes 8–15: amount (u64 LE)
- Bytes 16–23: interval (u64 LE)
- Bytes 24–31: plan_seed ([u8; 8])

**Accounts**:

| # | Account | Signer | Writable | Description |
|---|---|---|---|---|
| 0 | Subscription PDA | | Yes | New account to be initialized |
| 1 | Subscriber | Yes | Yes | Pays rent (~0.002 SOL) |
| 2 | Merchant | | | Merchant wallet (unchecked) |
| 3 | System Program | | | Required for account creation |

## PDA derivation

The subscription PDA is derived deterministically:

```typescript
import { findSubscriptionPda, planSeedToBuffer } from "@recur/sdk";

const planSeedBuf = planSeedToBuffer("0100000000000000");

const [pda, bump] = findSubscriptionPda(
  subscriberPubkey,
  merchantPubkey,
  planSeedBuf,
);
```

Or via the client helper, which returns `{ pda, bump }`:

```typescript
const { pda, bump } = recur.deriveSubscriptionPda(
  subscriberPubkey,
  merchantPubkey,
  "0100000000000000",
);
```

Each subscriber–merchant–plan combination has exactly one subscription PDA.
Attempting to create a duplicate will fail with an "already in use" error.

## Example: React integration

```typescript title="useSubscribe.ts"
import { useConnection, useWallet } from "@solana/wallet-adapter-react";
import { Transaction } from "@solana/web3.js";
import { RecurClient } from "@recur/sdk";

export function useSubscribe() {
  const { connection } = useConnection();
  const { publicKey, sendTransaction } = useWallet();

  const subscribe = async (
    planId: string,
    merchantWallet: string,
    planSeed: string,
    amount: number,
    intervalSeconds: number,
  ) => {
    if (!publicKey) throw new Error("Wallet not connected");

    const recur = new RecurClient({
      rpcUrl: connection.rpcEndpoint,
      apiBaseUrl: process.env.NEXT_PUBLIC_RECUR_API_URL!,
    });

    const { subscriptionPda, instructions } = recur.buildSubscribeTransaction(
      publicKey,
      { planId, merchantWallet, planSeed, amount, intervalSeconds },
    );

    const tx = new Transaction().add(...instructions);
    const signature = await sendTransaction(tx, connection);
    await connection.confirmTransaction(signature, "confirmed");

    return { signature, subscriptionPda: subscriptionPda.toBase58() };
  };

  return { subscribe };
}
```

After the on-chain transaction confirms, call
[`registerSubscription`](/docs/sdk/subscriptions#register-an-on-chain-subscription)
with the subscriber JWT to record the subscription in the Recur database. The
Keeper's chain scanner will pick it up as a fallback if the API call is missed.

## Re-approving an exhausted delegation

Once `delegationCycles` worth of payments have been pulled, the SPL approve runs
out and the next `process_payment` will fail. The on-chain subscription is still
active — the subscriber just needs to re-delegate.

```typescript
const { instructions } = recur.buildReapproveTransaction(publicKey, {
  merchantWallet,
  planSeed,
  amount,
  delegationCycles: 12, // optional; defaults to 12
});

const tx = new Transaction().add(...instructions);
await sendTransaction(tx, connection);
```

This emits a single SPL `approve` instruction for `amount × delegationCycles`
against the existing subscription PDA — no on-chain subscription state is touched.

## Error cases

| Error | Cause |
|---|---|
| `InvalidAmount` | Amount is below $1.00 (1,000,000 base units) |
| `InvalidInterval` | Interval is 0 |
| Account already in use | Subscription PDA already exists for this subscriber–merchant–plan |
| Insufficient SOL | Subscriber needs ~0.002 SOL for PDA rent |
